## 🔖 1. Supported Versions

Only the following versions of Zencroo Studio are actively receiving security updates. Please ensure you are running a supported version before reporting.

| Version | Status | Support Level |
| :--- | :---: | :--- |
| **V1.0.1 (Stable)** | ✅ Supported | Full security updates & patches |
| < V1.0.1 (Pre-release) | ❌ Unsupported | No security updates |

> [!IMPORTANT]
> We strongly recommend always running the **latest stable release** to ensure you have all security patches applied.

## 🔍 2. Reporting a Vulnerability

> [!CAUTION]
> **Please do NOT report security vulnerabilities via public GitHub Issues.** Public disclosure before a fix is available can put all users at risk.

To report a vulnerability, please contact us through **private, direct email** only:

📧 **Security Team Email:** [blurotech.in@gmail.com](mailto:blurotech.in@gmail.com)

**Use the subject line:** `[SECURITY] Zencroo Studio - <Brief Description>`

### What to Include in Your Report
To help us triage and resolve the issue efficiently, please provide the following details:

*   **Description**: A clear description of the vulnerability and its potential impact.
*   **Affected Version**: The Zencroo Studio version where the issue was found.
*   **Steps to Reproduce**: A detailed, step-by-step guide to reliably reproduce the issue.
*   **Proof of Concept**: Code snippets, screenshots, or recordings that demonstrate the vulnerability (if applicable).
*   **Suggested Fix**: If you have a proposed solution or mitigation, please include it.

## 📢 3. Disclosure Policy

We follow a **Coordinated Vulnerability Disclosure (CVD)** policy:

1.  You report the vulnerability to us privately.
2.  We acknowledge receipt of your report within **72 hours**.
3.  We investigate, reproduce, and develop a fix.
4.  We release a patched version and notify you.
5.  You may publicly disclose the vulnerability **30 days after the patch is released**.

> [!NOTE]
> We credit all security researchers who responsibly report valid, in-scope vulnerabilities in our release notes (unless they prefer to remain anonymous).

## 🎯 4. Security Scope

The following are **in-scope** for security vulnerability reports:

*   **Data Integrity**: Issues that could cause Zencroo Studio to silently corrupt, misrepresent, or lose user data (logged CSV files, project files, etc.).
*   **Privilege Escalation**: Any vulnerability that allows the application to gain unintended system privileges.
*   **Remote Code Execution**: Any scenario where specially crafted serial data or project files could execute arbitrary code on the user's machine.
*   **Local File Access**: Unauthorized read or write access to files outside the application's intended scope.
*   **Denial of Service**: Inputs that reliably crash the application or render it completely non-functional.

---

## 🚫 5. Out of Scope

The following are **not considered** security vulnerabilities:

*   Issues in operating system versions that are no longer supported by Microsoft.
*   Vulnerabilities in third-party libraries that have not been adapted into Zencroo Studio.
*   Reports generated by automated scanning tools without manual validation.
*   UI/UX issues, cosmetic bugs, or feature requests.
*   Issues that require physical access to the user's machine to exploit.
